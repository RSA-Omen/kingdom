Gekko Tracks

Tracks is the field-service surface of the platform. It owns the lifecycle of a job from creation through assignment, on-site capture, and reconciliation back into Pronto Xi.

Internally, Tracks is a thin orchestration layer over three older systems: the dispatch tables in Pronto, the technician roster in Almanac, and the parts ledger in Quarry. The Tracks API normalizes these into a single resource — the Job — and emits events on every transition.

Source of truth.

The Tracks Job resource is canonical. Do not write to JOB_HDR directly — use the API or go through Bender.

Architecture

Tracks runs as a Node service behind The Interceptor. Every read goes through Postgres replicas; writes go through the primary and produce an event on the tracks.jobs topic.

Data model

A job has four lifecycle states. Transitions are recorded as immutable rows in job_events; the current state is a materialized projection.

type JobState =
  | "draft"      // created, not assigned
  | "dispatched" // assigned to a tech
  | "on-site"    // tech checked in
  | "closed";    // reconciled in Pronto

Dispatch loop

The dispatcher polls open jobs every 30s, scores candidate technicians against the SLA + travel cost + skill match, and offers the top match. Offers expire in 90s.

  • Scoring happens in-memory; the table is bounded by the size of the active roster (~120).
  • Offers are stored in Redis with a TTL. Expiration is a Redis keyspace event.
  • Fallback jobs that fail three offers escalate to dispatch_review.

Integrations

Pronto Xi

Job closure must reconcile parts, time, and travel against Pronto's service order. Tracks does this through Bender — the keystroke harness — because no API surface exists for the SERV_ORD_CLS screen.

Bender lag during EOM.

Pronto's nightly EOM batch holds SERV_ORD_CLS for ~40 minutes. Tracks queues closures and replays them after the lock clears — expect delayed reconciliation between 23:00 and 23:45 SAST on the last day of each month.

Bender bridge

Each closure produces a Bender script run keyed by job.id. Run results stream back on the bender.results topic and update the job's reconciliation field. See Bender for the script grammar.

Runbook

Standard incident playbook. The dispatcher is the most failure-prone surface — start there.
