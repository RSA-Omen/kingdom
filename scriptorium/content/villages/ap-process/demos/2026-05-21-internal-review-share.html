Internal review share — collaborative clarification flow

Today, when Finance can't tell what an invoice is for or whether it's legitimate, they print it, scribble a note, and leave it on someone's desk. This is the digital version of that — same hand-off, with the conversation captured against the invoice forever.

Three flavours of review request

The same hand-off pattern, but the structure of the reply differs by intent. Worth distinguishing because flavour 3 — coding help — has the biggest payoff if the reply is structured rather than free text.

Flavour Finance is asking Reviewer is Reply shape
"What is this?" Identification / context Person likely to have ordered it Free-text comment
"Is this legit?" Approval / fraud check Dept head, project manager approve / reject + reason
"How do I code it?" GL / project allocation Cost-centre owner Structured: GL + project + split %

Design choices worth deciding before building

  • Internal only, or also external reviewers? Contractor PMs might need a magic-link email instead of a login.
  • One reviewer or many? Sequential approval chain vs. parallel "anyone can answer."
  • SLA / nudge? Auto-Telegram the assignee after 2 days; escalate after 5.
  • Annotation depth. Sticky-note comments are cheap; freehand drawing on the PDF needs a real annotation layer (PSPDFKit, react-pdf + canvas).
  • Scope creep risk: once you have an inbox + thread model, it stops being AP-specific — same pattern could carry expense queries, PO approvals, vendor onboarding.

Smallest useful first slice: assignee + free-text thread + email notification + Approve/Reject/Comment buttons + audit log. No PDF drawing, no structured GL replies, no external reviewers. Covers ~80% of what the paper-and-desk flow does today.
