Kingdom Dispatches — a publishing channel for the king

A formal home for the HTML reports Claude has been mailing to the king. Adopted as the Kingdom's third communication standard, alongside Telegram and Telegraph.

In plain English

You've been keeping a page at gvdi-30:8095 where Claude drops HTML files when something needs more than a chat reply. We're making that official: it's now a recognised part of the Kingdom and one of four ways the Kingdom talks to you.

What you get

  • A landing page at gvdi-30:8095/Kingdom/ that lists every dispatch, newest first.
  • A house style — Void Teal, the same palette as the dashboard — so every dispatch looks like part of the Kingdom, not a random web page.
  • A rule: every dispatch leads with "In plain English" before "Technical detail". You should never have to read jargon to find out what's happening.
  • Each dispatch is a single, self-contained HTML file. You can save one, forward it, or print it — it'll still look right.

Where it fits

Four channels, four jobs:

ChannelJobWhen
ChatQuick answers, working togetherLive
DispatchSubstantial proposal, audit, comparisonOn-demand
Telegram"King, you need to act"Event-driven
TelegraphMorning paperDaily

Dispatches are the non-action publishing channel. They exist so a Claude session never has to choose between dumping 800 lines into the chat and losing the thinking entirely.

What changes for you

Almost nothing. When something deserves to be read more than once, I'll publish a dispatch and link you to it instead of typing a wall of text. Same URL you've been using. The first thing you'll notice is that the page now has an index and a house style.

The orphan container situation

A reports-server container is currently running on the host, but its compose project lives at ~/projects/AP-processing/.claude/worktrees/wizardly-lehmann-fefb2a/infrastructure/reports-server/ — a worktree that no longer exists. The container is functionally orphaned: still serving, but unmanaged.

This PR moves the canonical compose+config into capital/dispatches/infrastructure/. After merge, run:

docker stop reports-server && docker rm reports-server
cd ~/Kingdom/capital/dispatches/infrastructure
docker compose up -d
curl http://localhost:8095/health  # → {"status":"healthy",...}

Mount source (~/reports/) is unchanged, so no published content is lost.

Design tokens

The dispatch template inlines the same Void Teal token set as capital/dashboard/app/globals.css: #050510 page, #81e6d9 accent, rgba(8,8,32,0.85) surfaces with backdrop-blur, isometric triangle hex grid (48×84 SVG, 0.7px strokes at 12% alpha). Inter for body, JetBrains Mono for meta and code.

CSS is inlined per dispatch — no shared stylesheet. Tradeoff: ~6 KB duplicated per file, in exchange for self-contained artifacts that survive forwarding, archiving, and design-system drift.

The two-audiences contract

Every dispatch has exactly two <section class="audience"> blocks in this order: audience-plain first (teal left border, "In plain English" header), audience-technical second (tertiary left border, "Technical detail" header). This is enforced socially, not in code — but the template only has these two slots, so deviating takes deliberate work.

Authoring flow

  1. Copy templates/dispatch.html.tpl → published/YYYY-MM-DD-<slug>.html.
  2. Replace {{TITLE}}, {{DATE_HUMAN}}, {{DATE_ISO}}, {{DISPATCH_KIND}}, {{LEDE}}, {{PLAIN_BODY}}, {{TECHNICAL_BODY}}, footer metadata.
  3. Prepend a row to published/index.html <table> (newest first).
  4. Run infrastructure/publish.sh — rsyncs published/ → ~/reports/Kingdom/.
  5. Commit. Visible at http://gvdi-30:8095/Kingdom/.

Out of scope (deliberately)

  • No build pipeline. Dispatches are hand-authored HTML. If volume grows, revisit — for now, a one-page template plus a sync script is enough.
  • No auth. Reports server is intranet-only (gvdi-30). Same trust boundary as before.
  • No RSS / email push. Dispatches are pull-based by design. Telegram and Telegraph handle push.
  • No MCP tool for "publish a dispatch." Possible follow-up if Claude sessions want to publish without filesystem access, but day-one this lives entirely in source control.
