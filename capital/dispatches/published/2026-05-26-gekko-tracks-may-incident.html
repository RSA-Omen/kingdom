Gekko Tracks — May 2026 Incident Report

Village: Gekko Tracks Date: 26 May 2026 Author: Claude (Gekko Tracks session)
In plain English

What went wrong

When Finance released the May batch of credit card transactions, most cardholders opened their coding screen and found their transactions already had the wrong descriptions and project codes filled in — codes that belonged to other months or other people. They had to code over the top of incorrect pre-filled answers, which was confusing and slowed the whole process down.

A second, separate problem was that one copy of the May batch was accidentally created using test data from January instead of the real May bank export. That gave Michael Brown, Eloise Earl, and Debbie Stephenson fake transactions to code.

On top of that, when managers went to review submitted batches, some transactions already showed "manager approved" — even though no manager had touched them.

Who was affected

17 cardholders across the May cycle had stale pre-fills: Nigel Grigg (52 transactions), Clint Torney (25), Matthew Conroy (17), Kate Rouhan (17), Mark Donadio (15), and 12 others. 3 cardholders received entirely fake transactions from the January test import (Michael Brown, Eloise Earl, Debbie Stephenson).

What was fixed

  • All stale pre-fills cleared — 156 transactions reset to blank across 17 open batches. Work already done by Nigel (18 rows), Kate (15 rows), and Simon (6 rows) was preserved.
  • The AI prediction button was wired up — it existed in the code but was never connected to the screen. Cardholders can now click "Auto Predict" to get suggestions instead of seeing wrong pre-fills.
  • The system now blocks anything from forcing a transaction into "manager approved" status except the manager's own approval button.
  • One stale "already approved" transaction in Aimee Riley's submitted batch was corrected.

What still needs watching

  • Eloise Earl and Debbie Stephenson's batches contain UAT (test) transactions from January — these need to be voided and replaced with their real May transactions once the correct May data is re-imported for those cardholders.
  • The January test finance batch (batch 129) should be archived to prevent further confusion.

Root cause analysis

RC-1 — Stale Classification rows from recovered.sql High Fixed

recovered.sql inserted Classifications with real coding data but privileged statuses (manager_approved) and no last_updated_at timestamp. When Finance released batch 128 and create_cardholder_batch ran, it found these pre-existing rows and only wrote batch_id — carrying description, project, cost_category, source, and status forward untouched.

Fix: cardholders.py:651 now resets all coding fields and status to unclassified/NULL on every batch attachment, unless the row is already user_confirmed or manager_approved (defensive guard for edge cases).

RC-2 — UAT import used in production finance batch High Needs action

Finance batch 129 was created pointing at import_job_id = 1 (test-uat-import.csv). There is no guard preventing a finance batch from being created against an archived or test import job. Michael Brown's batch (131) was already completed and submitted with UAT transactions; Eloise Earl (130) and Debbie Stephenson (132) still have open batches with 9 and 44 UAT transactions respectively.

Fix needed: Add validation in the finance batch creation endpoint to reject archived import jobs. Archive batch 129 and replace the affected cardholder batches with real May transactions.

RC-3 — Classification endpoint accepted privileged statuses Medium Fixed

PUT /api/classifications/:id accepted any value for status including manager_approved and rejected. source was always stamped "user" regardless. This allowed a transaction in Aimee Riley's open batch to be set to manager_approved at 11:40:16 before she submitted at 11:41:38, causing the manager to see an anomalous status on review.

Fix: The endpoint now raises HTTP 403 if payload.status is manager_approved or rejected. Those statuses are only writable by the manager approval endpoint.

RC-4 — Auto Predict dead code Low Fixed

runAutoPredict() in CardholderDashboard.tsx was fully implemented and wired to the backend (POST /api/cardholders/:id/batches/:bid/auto-predict) but never called from any UI element. The "Use Predictions" button only toggled the display of ML suggestion chips — it did not write predictions to the database. Cardholders had no way to trigger the prediction engine from the UI.

Fix: Added "Auto Predict" button to the batch toolbar. 1,339 manager_approved history rows are available for V2 matching (recurring, vendor, fuzzy).

RC-5 — No audit trail on manager batch approval Low Needs action

managers.py approve_batch sets all classifications to manager_approved with no audit_logs entry. When investigating how rows reached manager_approved status, there was no log to query — the source had to be inferred from source and last_updated_at values on the Classification rows themselves.

Prevention plan

P-1 — Retire or sanitise recovered.sql

The file should either be deleted (its data is now in production) or have all status and source values stripped before it can ever be run again. Running it a second time would re-insert the same stale rows via INSERT OR IGNORE, bypassing the new batch creation reset (which only fires on batch attachment, not on Classification row creation). Recommend: move to backend/archive/ with a README warning.

P-2 — Guard finance batch creation against archived import jobs

Add a validation check in POST /api/finance/batches: reject any import_job_id whose status is not "completed", or whose file_name contains "uat" / "test" / "demo". Optionally show a confirmation modal in the Admin Finance UI when releasing a batch.

P-3 — Audit log on manager batch approval

managers.py:approve_batch should write an audit_logs row recording manager_id, batch_id, and timestamp. Same for reject. This makes the approval trail queryable without having to reverse-engineer Classification source fields.

P-4 — Staging environment for UAT data

UAT/demo imports should be run against a separate SQLite file (ccc.demo.db) or a staging deploy, never against ccc.db. The existing CCC_DB_URL env var already supports this — it just needs policy enforcement and a separate startup script for demo sessions.

P-5 — Classification status state machine

The Classification status field currently accepts arbitrary strings with no transition guard. A simple allowed-transitions map should be enforced at the service layer:

  • unclassified → predicted, user_confirmed
  • predicted → user_confirmed, unclassified
  • user_confirmed → manager_approved, rejected, unclassified
  • manager_approved → (terminal — no further transitions except by admin)
  • rejected → unclassified (cardholder re-codes)

P-6 — Smoke test on finance batch release

After create_cardholder_batch runs, assert that all newly attached Classification rows have status = "unclassified" and description IS NULL. Fail fast rather than silently carry stale data forward.
