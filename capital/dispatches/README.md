# Kingdom Dispatches

The Kingdom's on-demand HTML publishing surface. When a Claude session has something for the king that deserves more than a chat reply — a proposal, an audit, a side-by-side comparison, anything worth re-reading — it ships as a **dispatch**.

**Live at:** `http://gvdi-30:8095/Kingdom/` (intranet) or `http://localhost:8095/Kingdom/` (on the host).

---

## Where this fits

Four communication channels, four jobs:

| Channel | Job | Cadence | Audience |
|---|---|---|---|
| **Chat** | Quick answers, ongoing work | Live | Whoever's in the session |
| **Dispatch** | Substantial proposal, review, comparison | On-demand | King — re-readable, forwardable |
| **Telegram** | "King, you need to act" | Event-driven | King — action-only |
| **Telegraph** | Morning paper | Daily | King (plus Brief edition, Weekly edition) |

Dispatches are the **non-action** publishing channel. They exist so a Claude session never has to choose between dumping 800 lines into chat and losing the thinking entirely.

See `docs/COMMUNICATION.md` for the canonical standard.

## Layout

```
capital/dispatches/
├── README.md                       ← you are here
├── infrastructure/
│   ├── docker-compose.yml          ← nginx container that serves ~/reports/
│   ├── nginx.conf                  ← server block (autoindex, /health)
│   └── publish.sh                  ← sync published/ → ~/reports/Kingdom/
├── templates/
│   └── dispatch.html.tpl           ← Void Teal dispatch skeleton (copy + edit)
└── published/
    ├── index.html                  ← landing page (lists dispatches reverse-chrono)
    └── YYYY-MM-DD-<slug>.html      ← one self-contained file per dispatch
```

**Source of truth:** `published/` (committed to git).
**Serving directory:** `~/reports/Kingdom/` (regenerated by `publish.sh`).

The serving directory is a build artifact — don't edit it by hand.

## Authoring a new dispatch

1. Copy `templates/dispatch.html.tpl` to `published/YYYY-MM-DD-<slug>.html`.
2. Fill in `<title>`, the header date, the body, and the metadata footer.
3. Update `published/index.html` — add a new row at the top of the list (newest first).
4. Run `infrastructure/publish.sh`. It rsyncs `published/` → `~/reports/Kingdom/`.
5. Commit. The king will see it at `http://gvdi-30:8095/Kingdom/`.

### The dispatch contract

Every dispatch follows the **two-audiences** pattern (this is a hard rule per `docs/COMMUNICATION.md`):

- **In plain English** — what's happening, why it matters, what the king should decide. No jargon.
- **Technical detail** — paths, commands, code, tradeoffs. For when the king wants to verify or hand off.

The two sections appear in that order. Plain English is never optional.

### Self-contained, always

Every dispatch is a single HTML file with inline CSS. No shared stylesheets, no external scripts. This means:

- The king can save one file and read it offline.
- The king can forward one file to a colleague.
- A dispatch from 2026-05 still renders correctly in 2028 even if the design system has moved on.

The cost (a few KB of duplicated CSS per file) is worth it.

## Infrastructure

The reports server is a small nginx container that serves `~/reports/` as a static directory tree with autoindex enabled. It currently hosts the Kingdom's dispatches at `/Kingdom/`; other projects may publish to sibling subdirectories (e.g. `/AP-processing/`).

**Bring up:**
```bash
cd capital/dispatches/infrastructure
docker compose up -d
```

**Health check:** `curl http://localhost:8095/health` → `{"status":"healthy"}`

**Note on the existing container:** A `reports-server` container is currently running from an orphaned compose project at `~/projects/AP-processing/.claude/worktrees/wizardly-lehmann-fefb2a/` (the worktree itself is gone). After this PR merges, take down the orphan and bring it back up from this directory:

```bash
docker stop reports-server && docker rm reports-server
cd ~/Kingdom/capital/dispatches/infrastructure
docker compose up -d
```

The mount source (`~/reports/`) is the same, so no published content is lost.

## Why HTML and not Markdown?

Markdown renders inconsistently across clients (GitHub vs. Slack vs. a phone vs. a printout). HTML renders the same everywhere. For a channel whose entire purpose is "something the king will want to re-read or forward," consistency wins.

The dispatch template uses the Kingdom's **Void Teal** palette — same tokens as the dashboard — so dispatches look like part of the platform, not like random web pages.
